📊 Permission Scanner Power Automate Flow

Version: 1.0 | Created: February 2026 | Flow Name: ScanMeetingFolderPermissions

🎯 Overview

The Permission Scanner flow automates the discovery and recording of SharePoint permission inheritance breaks (broken permissions) for meeting folders. It scans all items (files and folders) within a meeting's SharePoint folder, identifies items with unique permissions, and records detailed permission information in Dataverse.

Key Capability: Combines multiple permission levels per user/group (e.g., "Full Control, Edit") instead of recording only the first permission.

📋 Prerequisites

Dataverse Tables Required:

  • crad9_meeting - Must have columns: crad9_folderurl, crad9_level
  • crad9_scan - Created by create_permission_scanner_tables.py (12 columns)
  • crad9_itempermission - Created by create_permission_scanner_tables.py (11 columns)

Connections Required:

  • Microsoft Dataverse - For creating/updating scan and permission records
  • SharePoint Online - For querying items and permissions via REST API

🔄 Flow Architecture

High-Level Flow: 1. TRIGGER (Manual Button) ↓ 2. TRY SCOPE ├─ Get Meeting Record (Dataverse) ├─ Check if FolderURL exists ├─ Create Scan Record (status: In Progress) ├─ Parse SharePoint Site URL & Folder Path ├─ Fetch All Items from SharePoint (REST API) ├─ Filter: Items with HasUniqueRoleAssignments = true ├─ Filter: Items at depth >= meeting.level ├─ FOR EACH filtered item: │ ├─ Get Item RoleAssignments (permissions) │ ├─ FOR EACH permission assignment: │ │ ├─ Combine all RoleDefinitionBindings (Join with ", ") │ │ ├─ Create crad9_itempermission record │ │ └─ Increment permission counter │ └─ Increment item counter ├─ Update Scan Record (status: Completed, counters updated) └─ (If FolderURL missing) → Terminate with error ↓ 3. CATCH SCOPE (on failure) └─ Update Scan Record (status: Failed, error message)

5.1 - Parse_Folder_URL (Split URL into Array)

Action Type: Compose

Actual Power Automate Expression: @split(outputs('Get_Meeting_Record')?['body/crad9_folderurl'], '/')
Expression Breakdown - Part by Part
Expression Part What It Means
outputs('Get_Meeting_Record') Get the output from the "Get_Meeting_Record" action (the previous step that fetched the meeting from Dataverse)
?['body/crad9_folderurl'] From that output, safely access the crad9_folderurl field from the response body
The ? means "safe navigation" - won't crash if the field is missing
split(..., '/') Split the URL string by the '/' character into an array of parts
How Split Function Works
Input (Full URL from Meeting Record): "https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test/Meetings/Q1 2026" Split Process: The function finds every '/' character and breaks the string at those points Original: https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test/Meetings/Q1 2026 ↓ ↓ ↓ ↓ ↓ ↓ Split at: / / / / / / Result: Part0 Part1 Part2 Part3 Part4 Part5 Part6
Output Array - Element by Element
Index Value Explanation Where It's Used
[0] "https:" The protocol part before the first '/'
Original: https///...		 Used in Set_Site_URL to rebuild the protocol
[1] "" (empty string) Between the two slashes in ':///'
Original: https:///...		 Not used (empty because of double slash)
[2] "ABCTest179.sharepoint.com" The domain name between '//' and '/sites'
Original: //ABCTest179.sharepoint.com/sites...		 Used in Set_Site_URL to rebuild the domain
[3] "sites" The SharePoint path segment
Original: .../sites/Permission...		 Used in Set_Site_URL (hardcoded as "/sites/" in expression)
[4] "Permission-Scanner-Test" The SharePoint site name
Original: .../Permission-Scanner-Test/Meetings...		 Used in Set_Site_URL to complete the site URL
[5] "Meetings" The document library name
Original: .../Meetings/Q1 2026		 Used in Set_Folder_Path (part of folder path)
[6] "Q1 2026" The folder name (end of URL, no slash after it)
Original: .../Q1 2026		 Used in Set_Folder_Path (part of folder path)

5.2 - Set_Site_URL (Reconstruct Site URL)

Action Type: Compose

Actual Power Automate Expression: @{outputs('Parse_Folder_URL')[0]}//@{outputs('Parse_Folder_URL')[2]}/sites/@{outputs('Parse_Folder_URL')[4]}
Expression Breakdown - Part by Part
Expression Part What It Means
@{outputs('Parse_Folder_URL')[0]} Get element at index [0] from the Parse_Folder_URL output
Result: "https:"
// Hardcoded literal string: two forward slashes
This reconstructs the double slash after the protocol
@{outputs('Parse_Folder_URL')[2]} Get element at index [2] from the Parse_Folder_URL output
Result: "ABCTest179.sharepoint.com"
/sites/ Hardcoded literal string
SharePoint always uses "/sites/" for site collections
@{outputs('Parse_Folder_URL')[4]} Get element at index [4] from the Parse_Folder_URL output
Result: "Permission-Scanner-Test"
How the URL is Reconstructed - Step by Step
Step 1: Start with array from Parse_Folder_URL ["https:", "", "ABCTest179.sharepoint.com", "sites", "Permission-Scanner-Test", "Meetings", "Q1 2026"] └─[0]─┘ └[1]┘ └──────────[2]──────────┘ └─[3]─┘ └────────[4]───────────┘ └──[5]──┘ └───[6]──┘ Step 2: Extract needed elements [0] → "https:" [2] → "ABCTest179.sharepoint.com" [4] → "Permission-Scanner-Test" Step 3: Combine with hardcoded separators @{[0]} // @{[2]} /sites/ @{[4]} ↓ ↓ ↓ ↓ ↓ "https:" "//" "ABCTest179.sharepoint.com" "/sites/" "Permission-Scanner-Test" Step 4: Result (concatenated string) "https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test"
Why We Skip Elements [1] and [3]
Skipped Element Value Why It's Skipped
[1] "" (empty string) This was created by the double slash in "https://". We hardcode "//" instead of using this empty element.
[3] "sites" We hardcode "/sites/" as a literal string. This makes the expression more readable and ensures consistency.
Why This Expression Pattern Was Chosen
Alternative Approach (NOT Used):
❌ BAD: @{[0]}/@{[1]}/@{[2]}/@{[3]}/@{[4]} Result: "https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test" Problem: Uses empty string [1], harder to read, more elements to track
Actual Approach (USED):
✅ GOOD: @{[0]}//@{[2]}/sites/@{[4]} Result: "https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test" Benefits: • Skips empty string [1] by hardcoding "//" • Hardcoded "/sites/" makes it clear this builds a SharePoint site URL • Only uses 3 array elements instead of 5 • More maintainable and self-documenting

Example 1: Different SharePoint Tenant
Original URL: "https://contoso.sharepoint.com/sites/Sales-Reports/Documents/Q4" After Parse_Folder_URL: [0] = "https:" [1] = "" [2] = "contoso.sharepoint.com" [3] = "sites" [4] = "Sales-Reports" [5] = "Documents" [6] = "Q4" Set_Site_URL builds: @{[0]} // @{[2]} /sites/ @{[4]} ↓ ↓ ↓ ↓ ↓ "https:" "//" "contoso.sharepoint.com" "/sites/" "Sales-Reports" Result: "https://contoso.sharepoint.com/sites/Sales-Reports"

Where This URL Is Used
The reconstructed Site URL is used as the "dataset" parameter in SharePoint HTTP requests:
Step 7 - Send_HTTP_to_SharePoint: { "dataset": "@outputs('Set_Site_URL')", "parameters/method": "GET", "parameters/uri": "_api/web/lists/GetByTitle('Meetings')/items?..." } Full request becomes: https://ABCTest179.sharepoint.com/sites/Permission-Scanner-Test/_api/web/lists/... └──────────────── Site URL from Set_Site_URL ──────────────────┘└── Relative URI ──┘
Why separate Site URL from URI?
  • SharePoint connector requires "dataset" (site URL) and "uri" (relative path) as separate parameters
  • Allows the same URI to be reused across different SharePoint sites
  • Power Automate handles authentication per site URL

5.3 - Set_Folder_Path (Extract Folder Path)

Action Type: Compose

Actual Power Automate Expression: @join(skip(outputs('Parse_Folder_URL'), 5), '/')
Expression Breakdown - Part by Part
Expression Part What It Means
outputs('Parse_Folder_URL') Get the array output from Parse_Folder_URL action
Example: ["https:", "", "ABCTest179.sharepoint.com", "sites", "Permission-Scanner-Test", "Meetings", "Q1 2026"]
skip(..., 5) Skip the first 5 elements of the array
This removes the site URL parts [0] through [4], keeping only folder path elements
join(..., '/') Join the remaining array elements with '/' separator
Converts array back into a single path string
How Skip() Function Works - Step by Step
Step 1: Start with the full array ["https:", "", "ABCTest179.sharepoint.com", "sites", "Permission-Scanner-Test", "Meetings", "Q1 2026"] └─[0]─┘ └[1]┘ └──────────[2]──────────┘ └─[3]─┘ └────────[4]───────────┘ └──[5]──┘ └───[6]──┘ │ │ │ │ │ │ │ └───────┴───────┴────────────────────────┴───────┴──── skip(5)─────────────┘ │ ↓ │ These are These are REMOVED KEPT Step 2: skip(array, 5) removes first 5 elements Original: [0] [1] [2] [3] [4] [5] [6] "https:" "" "ABCTest179.sharepoint..." "sites" "Permission-Scanner..." "Meetings" "Q1 2026" └───────────────── SKIP THESE (site URL parts) ──────────────────┘ └── KEEP THESE ──┘ After skip(5): ["Meetings", "Q1 2026"] └───[0]───┘ └───[1]──┘ (new indexes after skip) Step 3: join() combines remaining elements with '/' Array: ["Meetings", "Q1 2026"] └────┬────┘ └───┬───┘ │ │ └─────┬──────┘ ↓ join with '/' ↓ Result: "Meetings/Q1 2026"
Why Skip Exactly 5 Elements?
SharePoint URL Structure is Predictable:
Index Always Contains Purpose
[0] Protocol (https:) These 5 elements always make up the SharePoint Site URL

They're constant for all folders within the same site, so we skip them to get just the folder path
[1] Empty string (from //)
[2] Tenant domain
[3] Literal "sites"
[4] Site name
[5+] Library and folder names These elements form the folder path within the site
Variable length depending on folder depth

Step 9: Filter - Broken Permissions Only

Action: Query (Filter Array)

Condition: HasUniqueRoleAssignments = true

This filter removes all items that inherit permissions from parent folders, leaving only items with unique permission assignments (broken inheritance).

Step 10: Filter - Level (Depth) Check

Action: Select (Transform Array)

Action Name: Filter_By_Level

Actual Power Automate Expression

From: @body('Filter_Broken_Permissions') Select: @if( greaterOrEquals( sub( length(split(item()?['FileRef'], '/')), length(split(outputs('Set_Folder_Path'), '/')) ), outputs('Get_Meeting_Record')?['body/crad9_level'] ), item(), null )

Expression Breakdown

Part Expression What it Does
1. Item Depth length(split(item()?['FileRef'], '/')) Count how many "/" separators in item path
Example: "/sites/Site/Meetings/Q1/File.pdf" → 5
2. Folder Depth length(split(outputs('Set_Folder_Path'), '/')) Count how many "/" separators in folder path
Example: "Meetings/Q1 2026" → 2
3. Relative Depth sub(itemDepth, folderDepth) Calculate depth relative to meeting folder
Example: 5 - 2 = 3
4. Compare Level greaterOrEquals(relativeDepth, meeting.level) Check if item is deep enough
Example: 3 >= 2 → true
5. Return Value if(condition, item(), null) If true: return item
If false: return null (filtered out)

Understanding the Level Setting

What does "Level" mean?

The crad9_level field on the meeting record controls the minimum relative depth of items to scan.

Level Value What Gets Scanned Use Case
0 Everything (all items at any depth) Scan all items including root folder itself
1 Items at depth 1 or deeper Skip root folder, scan everything inside
2 Items at depth 2 or deeper Skip files in root, only scan subfolders and their contents
3 Items at depth 3 or deeper Only scan deep nested items (e.g., Meetings/Q1/Jan/File.pdf)

Visual Example: Folder Structure with Level Filtering

Meeting Folder: /sites/Permission-Scanner-Test/Meetings/Q1 2026 Folder Path: "Meetings/Q1 2026" (depth = 2) 📁 Meetings/Q1 2026/ ← Relative Depth: 0 ├─ 📄 RootFile.pdf ← Relative Depth: 1 ├─ 📁 Subfolder1/ ← Relative Depth: 1 │ ├─ 📄 File1.docx ← Relative Depth: 2 │ └─ 📁 Nested/ ← Relative Depth: 2 │ └─ 📄 File2.xlsx ← Relative Depth: 3 └─ 📁 Subfolder2/ ← Relative Depth: 1 └─ 📄 File3.pdf ← Relative Depth: 2 Level = 0: All items scanned (7 items) Level = 1: RootFile.pdf, Subfolder1/, File1.docx, Nested/, File2.xlsx, Subfolder2/, File3.pdf (7 items) Level = 2: File1.docx, Nested/, File2.xlsx, File3.pdf (4 items) Level = 3: File2.xlsx (1 item)

Why Use Select Instead of Filter?

Technical Note: Power Automate uses Select action (not Filter) because:
  • Select transforms each item (returns item or null)
  • Filter only accepts true/false conditions (can't return null)
  • Null values are then filtered out in the "Apply to each" loop using the "Condition_Not_Null" check
Two-Step Filtering:
  1. Select (Filter_By_Level): Returns item or null based on depth
  2. Foreach (Condition_Not_Null): Skips null entries, processes valid items

{ "d": { "results": [ { "Member": { "Title": "John Doe", "PrincipalType": 1 // 1=User, 8=Group }, "RoleDefinitionBindings": { "results": [ {"Name": "Full Control"}, {"Name": "Edit"} ] } } ] } }

11.5 - Loop - Process Each Permission Assignment

Action: Apply to each (Nested foreach)

11.5.1 - Select Permission Names

Action Type: Select

Purpose: Extract only the permission names from RoleDefinitionBindings

Actual Expression: From: @items('Apply_to_each_Permission')?['RoleDefinitionBindings']?['results'] Select: @item()?['Name'] What it does: Transforms an array of role objects into an array of just their names Input (RoleDefinitionBindings): [ {"Name": "Full Control", "Id": 1073741829, ...}, {"Name": "Edit", "Id": 1073741830, ...} ] Output (Permission Names): ["Full Control", "Edit"]
11.5.2 - Join Permissions (KEY FEATURE)

Action Type: Join

Purpose: Combine multiple permission levels into a single comma-separated string

Actual Expression: From: @body('Select_Permission_Names') JoinWith: ", " What it does: Joins array elements with ", " separator (comma + space) Input (Array): ["Full Control", "Edit"] Output (Joined String): "Full Control, Edit" Stored in: crad9_permissionlevel field
Critical Feature: This step combines multiple permission levels into a single string, addressing the user requirement to "combine the permissions and store them as one, not just the first."

Why this matters:

  • A user can have MULTIPLE permission levels on the same item (e.g., member of both "Editors" and "Viewers" groups)
  • Previous approach: Only stored first permission found
  • Current approach: Stores ALL permissions as "Full Control, Edit, Read"
  • Provides complete audit trail of all permission grants

🐛 Troubleshooting

Issue: "Meeting does not have a folder URL"

Cause: crad9_folderurl field is empty

Solution: Run CreateMeetingSPFolder flow first to populate crad9_folderurl

Issue: "List 'Meetings' does not exist"

Cause: SharePoint document library not found

Solution: Verify "Meetings" library exists at the SharePoint site

Issue: No items returned from SharePoint

Possible Causes:

  • Folder path incorrect (check URL encoding)
  • No items in folder
  • Permission issue (flow user can't access items)

Solution: Check scan record error message, verify folder URL manually

Issue: Permission records created but empty permission level

Cause: RoleDefinitionBindings array is empty

Solution: Check SharePoint permissions - item may have no explicit permissions granted

Issue: Flow times out on large folders

Cause: Too many items (>500) or too many permission assignments

Solution: Implement pagination or reduce level setting to scan fewer items
