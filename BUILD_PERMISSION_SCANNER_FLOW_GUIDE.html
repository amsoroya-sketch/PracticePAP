📋 Overview

Flow Name: ScanMeetingFolderPermissions

Purpose: Scan SharePoint folder permissions for a meeting, identify items with broken inheritance, and record permission details in Dataverse.

Trigger Type: Manual (Button) - accepts Meeting ID as input

Complexity: Advanced (29 actions, nested loops, error handling)

Estimated Build Time: 45-60 minutes

What This Flow Does:

  1. Receives a Meeting ID from manual trigger
  2. Fetches meeting details from Dataverse (including SharePoint folder URL)
  3. Creates a scan record to track the operation
  4. Parses the SharePoint URL to extract site and folder paths
  5. Queries SharePoint for all items in the meeting folder
  6. Filters items with broken inheritance (unique permissions)
  7. Filters items by configured depth level
  8. For each item with broken permissions:
    • Gets all permission assignments
    • Extracts permission names (Full Control, Edit, Read, etc.)
    • Combines multiple permissions into single string
    • Creates permission record in Dataverse
  9. Updates scan record with results and statistics
  10. Handles errors gracefully with try-catch pattern

⭐ Best Practices for Building This Flow

✓ 1. Use Scopes for Error Handling

Why: Scopes allow you to catch errors across multiple actions with a single try-catch pattern.

Try Scope ├─ Get_Meeting_Record ├─ Create_Scan_Record ├─ ... (all main logic) └─ Update_Scan_Complete Catch Scope (runs on Try Scope failure) └─ Update_Scan_Failed (set status to Failed)

Benefit: If ANY action fails, the Catch Scope runs and updates the scan record with error details.

✓ 2. Rename All Actions for Clarity

Why: Default names like "Get a row by ID" are unclear. Use descriptive names like "Get_Meeting_Record".

❌ Default Name ✅ Better Name Why Better
Get a row by ID Get_Meeting_Record Instantly clear what record you're getting
HTTP Send_HTTP_to_SharePoint Shows what API you're calling
Apply to each Apply_to_each_Item Shows what you're looping over

Benefit: Easier debugging, better documentation, clearer for other developers.

✓ 3. Use Compose Actions for Complex Expressions

Why: Breaking complex logic into Compose actions makes expressions reusable and easier to test.

Example: Instead of embedding URL parsing inline, create separate Compose actions:

  • Parse_Folder_URL → Split URL by '/'
  • Set_Site_URL → Rebuild site URL from array
  • Set_Folder_Path → Extract folder path

Benefit: Can test each step independently, easier to debug, clearer flow structure.

✓ 4. Initialize Variables at the Top

Why: Variables must be initialized before use, and keeping them together makes dependencies clear.

Order:

  1. Initialize_Items_Array (for collecting items)
  2. Initialize_Counter (items scanned)
  3. Initialize_Permissions_Counter (permissions recorded)
  4. Initialize_Broken_Counter (items with broken permissions)

Benefit: All variables visible at once, prevents "variable not found" errors.

✓ 5. Use Increment Variable Instead of Set Variable

Why: Increment Variable is atomic and prevents race conditions in loops.

❌ BAD: Set variable varCounter to add(variables('varCounter'), 1) Problem: In concurrent operations, value might be stale ✅ GOOD: Increment variable varCounter by 1 Benefit: Atomic operation, always accurate

✓ 6. Filter Before Looping to Reduce API Calls

Strategy: Filter data as early as possible to minimize iterations.

Flow order: 1. Get ALL items from SharePoint (500 max) 2. Filter: Only items with HasUniqueRoleAssignments = true 3. Filter: Only items at depth >= configured level 4. THEN loop over filtered subset

Benefit: If 500 items but only 10 have broken permissions, you loop 10 times instead of 500.

✓ 7. Use Select + Join for Combining Arrays

Use Case: A user can have multiple permission levels (e.g., "Full Control" AND "Edit").

Pattern:

Step 1 - Select: Extract just the 'Name' field from RoleDefinitionBindings Input: [{"Name": "Full Control", "Id": 123}, {"Name": "Edit", "Id": 456}] Output: ["Full Control", "Edit"] Step 2 - Join: Combine array into comma-separated string Input: ["Full Control", "Edit"] Output: "Full Control, Edit" Step 3 - Store: Save to crad9_permissionlevel field

Benefit: Complete audit trail - stores ALL permissions, not just the first one.

✓ 8. Use Safe Navigation Operator (?)

Why: Prevents null reference errors when accessing nested properties.

❌ BAD: outputs('Get_Meeting_Record')['body/crad9_folderurl'] Problem: Crashes if body doesn't exist ✅ GOOD: outputs('Get_Meeting_Record')?['body/crad9_folderurl'] Benefit: Returns null instead of crashing if body missing

🔧 Troubleshooting Guide

❌ Error: "Connector not found"

Cause: Connection not authorized

Fix:

  1. Click the action with the error
  2. Delete the old connection (if shown)
  3. Click "+ Add new connection"
  4. Sign in and create new connection

❌ Error: "404 Not Found" on SharePoint HTTP

Cause: Library or folder doesn't exist

Fix:

  1. Verify Meetings library exists in SharePoint
  2. Check folder URL in meeting record is correct
  3. Create folder manually if needed

❌ Error: "401 Unauthorized" on SharePoint HTTP

Cause: Insufficient permissions

Fix:

  1. Verify your account has Read access to SharePoint site
  2. Re-create SharePoint connection
  3. Ask site admin to grant permissions if needed

❌ Error: "The execution of template action 'Filter_By_Level' failed"

Cause: Invalid expression or null values

Fix:

  1. Check expression uses safe navigation: item()?['FileRef']
  2. Verify previous action returned valid data
  3. Check meeting.level is a number, not null

⚠️ Issue: No permissions found

Possible Causes:

  • No items in folder with broken inheritance
  • Level setting filters out all items
  • Folder is empty

To Debug:

  1. Check Send_HTTP_to_SharePoint output - any items returned?
  2. Check Filter_Broken_Permissions output - any items filtered?
  3. Check Filter_By_Level output - any items passed depth check?
  4. Try setting meeting.level to 0 to scan all depths
